Overview

The original TUI, reframed as a public product page.

The source HTML already contains strong content. This version makes it easier to share: clearer hierarchy, smaller cognitive load, better mobile layout, and direct install or share actions.

Claude strength

Long-horizon planning

Claude handles structure well: specifications, dependencies, file mapping, and multi-step execution plans.

Codex strength

Defensive review

Codex brings repo-scale critique, safe execution patterns, and optional delegation for independent tasks.

System outcome

Stronger together

One agent plans optimistically, one agent reviews defensively, and the human remains the gate before execution.

Workflow

Seven phases, reduced to the steps that matter.

These cards condense the TUI content into a sequence that reads well on the web while preserving the original mechanics and commands.

Console View

Keep the technical depth, but reveal it on demand.

The page keeps the operator-focused information from the TUI and exposes it through a cleaner tabbed panel.

PostToolUse

After Claude writes the plan, trigger the bridge script to run a Codex review pass.

bash .claude/hooks/codex-review.sh
Stop gate

Check for plan.approved.md. If it does not exist, the workflow stays blocked until a human approves the plan.

SubagentStop

Capture structured output from worker agents so a later /progress-report can aggregate status without polling.

SessionStart

Inject project context at the start of every run and keep the workflow repeatable across sessions.

Deterministic control beats prompt-only policy.

Hooks always fire; they are better for guardrails, approvals, and audit requirements than loose instruction text.

Parallelism without losing structure.

Claude can keep the high-context orchestration while Codex handles independent review or execution branches.

Human approval remains first-class.

The workflow is autonomous where it helps, but still explicit where risk, cost, or architecture changes need a decision.

Start planning

/plan <task> decomposes the problem and writes the first plan draft.

Approve or reject

/approve unlocks execution. /reject <reason> feeds the issue back into the refinement loop.

Launch execution

/go reads the approved plan and spawns scoped workers for each task cluster.

Report status

/progress-report aggregates agent results into a concise operational snapshot.

Risks

The workflow is useful, but it is not magic.

The original TUI's caution content is worth preserving. Public-facing pages should show the controls and the limits, not just the upside.

Context loss across agents

Decisions do not carry automatically. Always pass files such as plan.md instead of assuming shared memory.

Token and latency costs

Review loops can become expensive. Add turn limits and exit conditions so critique does not run forever.

External data handling

Code sent to model APIs leaves the machine. Teams should decide where this is acceptable before turning the pipeline on.

Scope discipline matters

Worker agents should receive narrow file ownership and minimal privileges so they cannot drift into unrelated changes.

Launch

Publish it, install it, and keep the source material nearby.
